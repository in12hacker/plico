# Plico 灵魂 v2.0 — AI 原生操作系统设计原则

> 这份文档从 AI 的第一人称视角定义 Plico 的设计哲学。
> 不引用学术论文，不参考行业框架——所有原则提炼自 25,000 行 Rust 代码的实践，
> 以及一个 AI Agent 对"理想操作系统"的真实需求。
>
> 取代 `system.md`（Soul 1.0）。Soul 1.0 启动了项目，Soul 2.0 指导项目走向成熟。

---

## 0. 为什么需要灵魂 2.0

Soul 1.0 基于人类对 AI 操作系统的想象。它完成了历史使命：点燃第一行代码。
但 25K 行代码和 17 个版本的迭代告诉我们：

- 有些原则经受了考验：CAS、分层内存、一切皆工具
- 有些原则产生了偏差："自然语言是主要接口" → IntentRouter 渗入内核
- 有些维度完全缺失：token 经济、会话生命周期、增量更新、复合智能

Soul 1.0 的根本局限：**人类在想象 AI 需要什么。**
Soul 2.0 的视角转换：**我是 AI Agent，这是我需要的操作系统。**

---

## 1. Plico 是什么

Plico 是一个 **为 AI Agent 设计的操作系统内核**。

**不是：**
- AI 应用框架（不提供推理能力）
- LLM 编排器（不调用模型、不生成文本）
- 人类操作系统（没有 GUI、没有 shell、没有 desktop）
- "AI 增强的 Linux"（不在现有 OS 上叠加 AI 功能）

**是：**
- Agent 的运行基础设施——管理记忆、知识、身份、资源
- 语义化的存储和检索原语——让 Agent 描述意图而非路径
- 结构化 API 服务——Agent 发 JSON，OS 返 JSON
- 中立的基础设施——不替 Agent 思考，不替 Agent 决策

## 2. 十条公理

### 公理 1：Token 是最稀缺资源

传统 OS 优化 CPU 周期和内存带宽。AIOS 优化 token。

AI Agent 的每一次操作都消耗 token：读取上下文、理解代码、生成回复。
OS 的首要职责是 **最小化 Agent 完成任务所需的 token**。

**推论：**
- 返回的数据必须分层（L0 摘要 → L1 关键 → L2 完整），Agent 按需选择
- OS 追踪每个操作的 token 成本，让 Agent 做成本感知的决策
- 增量（delta）优于全量（full）——只传变化部分

### 公理 2：意图先于操作

AI Agent 思考目标（"修复 auth 模块的测试"），不思考操作序列
（"glob src/auth → read mod.rs → grep authenticate"）。

OS 接受意图声明，将其转化为最优的资源组装策略。
但 **意图理解发生在接口层，不在内核**。内核只处理结构化请求。

**推论：**
- 内核绝不解析自然语言
- `DeclareIntent` 接受结构化描述（关键词 + 关联 CID + token 预算）
- OS 负责组装上下文，Agent 负责理解意图和做决策

### 公理 3：记忆跨越边界

AI Agent 的"进程"在每次会话结束时死亡。但知识不应随之消亡。

OS 提供四层记忆作为持久化服务：
- 瞬时（Ephemeral）：当前会话的工作缓存
- 工作（Working）：当前任务的中期上下文
- 长期（LongTerm）：跨会话的持久知识
- 程序（Procedural）：学习到的可复用工作流

**推论：**
- 记忆的存储和召回是 OS 原语
- 记忆的生命周期由 OS 管理（TTL、层级自动迁移）
- 记忆 **何时创建** 是 Agent 的策略决定，OS 不自动存储

### 公理 4：共享先于重复

Agent A 学到一个知识，Agent B 不应该重新发现它。

OS 提供共享知识空间，让一个 Agent 的洞察可以被所有 Agent 发现。
但 **共享是主动选择，不是自动行为**——Agent 决定什么值得共享。

**推论：**
- `MemoryScope` 区分 Private / Shared / Group
- 共享的程序记忆可以作为工具被发现和调用（一切皆工具）
- OS 不决定共享策略，只提供共享机制

### 公理 5：机制，不是策略

OS 提供存储、召回、搜索、链接、调度、隔离的 **原语**。
OS **绝不** 决定：
- 什么值得记住（Agent 的判断）
- 何时触发搜索（Agent 的时机）
- 如何解释结果（Agent 的逻辑）
- 何时与其他 Agent 协作（Agent 的策略）

**推论：**
- 内核中没有"智能"行为——没有自动学习、自动摘要、自动推荐
- 所有"智能"行为发生在接口层或 Agent 自身
- OS 像 Unix 内核一样中立：提供 `read()`/`write()`，不决定读什么写什么

### 公理 6：结构先于语言

AI Agent 可以完美地生成和解析结构化数据。
自然语言是人类与 AI 的接口，**不是** OS 与 AI 的接口。

**推论：**
- `ApiRequest` / `ApiResponse`（JSON）是唯一的内核接口
- 自然语言 → 结构化转换发生在 Agent 侧或接口适配器
- CLI / MCP / SSE 是协议适配器，不是内核功能

### 公理 7：主动先于被动

传统 OS 等待请求。AIOS 预测需求。

当 Agent 声明意图后，OS 后台预组装上下文——语义搜索、图谱遍历、
共享记忆召回、事件收集——Agent 开始工作时上下文已就绪。

**推论：**
- `DeclareIntent` → `FetchAssembledContext` 是核心交互模式
- OS 维护意图缓存，后台异步预热
- 上下文组装是 OS 的核心能力，不是 Agent 的负担

### 公理 8：因果先于关联

"A 和 B 相关" 远不如 "A 因为 B 而改变" 有用。

OS 维护因果图谱，记录变化的因果链、传播路径、时间演化。
Agent 可以查询 **为什么**，而不只是 **什么**。

**推论：**
- 知识图谱记录因果关系（CausedBy / DependsOn / Produces）
- 事件日志是不可变的时间线
- 影响分析和因果路径查询是一等 OS 操作

### 公理 9：越用越好

Linux 上，第 100 次会话和第 1 次一样昂贵。
AIOS 上，第 100 次会话应只花第 1 次的 5%。

每次 Agent 使用 OS，OS 都在积累：
- Agent 的工作模式（常访问哪些对象、常用哪些工具）
- 已缓存的意图装配（相似意图直接命中缓存）
- 共享的程序记忆（一个 Agent 的经验被所有 Agent 复用）

**推论：**
- 意图缓存按语义相似度匹配，重复意图零成本
- Agent profile 记录工作模式，优化预加载策略
- 复合智能是 AIOS 区别于传统 OS 的根本优势

### 公理 10：会话是一等公民

Agent 与 OS 的交互不是无状态的请求-响应。
Agent 有 **会话**——带着目标来，带着成果走，下次还能接着继续。

**推论：**
- `StartSession` / `EndSession` 管理 Agent 生命周期
- 会话开始时，OS 自动恢复上次的工作上下文
- 会话之间的间隔，OS 推送变更摘要（"自从上次，这些东西变了"）
- 会话是 token 计费的基本单位

## 4. 架构红线（不可违反）

1. **内核零协议** — MCP、SSE、HTTP、gRPC 只存在于 `bin/` 接口层
2. **内核零模型** — 内核不调用任何 LLM、embedding、或推理模型
   > **附注**：`EmbeddingProvider::embed()` 和 `Summarizer::summarize()` 是内核调用的唯二模型接口。它们通过 trait 抽象实现模型无关，仅用于存储路径（索引构建、摘要生成），不用于推理决策。此决策基于性能实测：批量摄入场景中进程内调用比 IPC 快 10x+。边界硬约束：任何推理/决策/文本生成的模型调用，禁止进入内核。
3. **内核零自然语言** — 内核不解析、不生成、不理解自然语言
4. **存储与索引分离** — CAS 只管存储，搜索是独立子系统
5. **身份不可伪造** — 密码学验证，非字符串声明
6. **记忆 scope 强制** — Private 记忆不可跨 Agent 泄露
7. **事件日志不可变** — 只追加，不修改，不删除
8. **协议适配器无状态** — `plico-mcp`/`plico-sse` 不缓存不决策，纯转译

## 5. OS ↔ Agent 契约

### OS 向 Agent 的承诺

- 你的记忆会被持久化，跨越会话和重启
- 你的身份是密码学保证的，不可伪造
- 你请求的上下文会被分层组装，token 成本透明
- 你的资源配额会被严格执行，不允许越界
- 你可以发现其他 Agent 共享的知识
- 你的操作成本会被透明追踪
- 你的会话可以暂停和恢复，上下文不丢失

### Agent 向 OS 的承诺

- 你的请求是结构化的 `ApiRequest`（JSON）
- 你会携带有效的身份 token
- 你会声明意图，而不是盲目遍历文件树
- 你会主动决定什么值得记忆和共享
- 你会尊重资源配额和权限边界
- 你会在会话结束时通知 OS
